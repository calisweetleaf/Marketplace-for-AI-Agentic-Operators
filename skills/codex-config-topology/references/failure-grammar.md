# FAILURE GRAMMAR — Codex Environment Control Plane

## Pre-Failure Signatures

- Codex gives baseline stock guidance even though injected doctrine exists.
- A BB7 tool other than exo is proposed as the first move after user input.
- `AGENTS.md` is read but `AGENTS.override.md` is ignored.
- `OPSEC.md` or `STYLE.md` are treated as optional lore instead of binding discipline.
- Terminal guidance suddenly becomes timid or sandboxed without evidence from rules or host policy.
- Permission claims contradict `default.rules` or the observed launch surface.
- A change lands in `.codex` but the real behavior still comes from the wrapper or host profile.

## False Success Patterns

### False success 1: The baseline constitution was loaded
`AGENTS.md` was read, but the active runtime override was not, so turn-order advice is still wrong.

### False success 2: Exo was used eventually
Exo appears in the loop, but not first, which means the intended routing posture was already violated.

### False success 3: Full terminal posture was honored
The advice sounded permissive, but it ignored the actual permission surfaces, wrapper path, or host runtime.

### False success 4: The doctrine files exist on disk
They exist, but nothing in the startup or first-mission path actually ingests them.

### False success 5: The right file was edited
The file is real, but it is not the owning surface for the behavior being changed.

## Recovery Protocols

1. Reconstruct the first-mission doctrine load order.
2. Identify whether the current turn entered BB7 through exo or through an incorrect side door.
3. Verify which file owns the behavior: `.codex`, injected doctrine, host profile, wrapper, `default.rules`, or `tool_manifest.json`.
4. Compare permission claims against the actual rules and host runtime.
5. Move the change to the owning surface and re-state startup-time versus task-time consequences.

## Rejection Criteria

Reject or pause a recommendation when any of these are true:

1. The recommendation assumes stock Codex defaults while evidence shows injected doctrine and custom orchestration.
2. The recommendation uses a non-exo BB7 entry path without explicit operator approval.
3. The recommendation adds imaginary permission constraints or claims unlimited authority without inspecting the real policy surfaces.

## Failure Questions

Ask these internally before proposing edits:

- were all injected doctrine files loaded for the first mission?
- what is the exact first BB7 move after user input?
- does this behavior belong to `.codex`, a doctrine file, a host profile, a wrapper, or a rules file?
- what permission evidence actually exists?
- is the current host the same one the operator is talking about?
